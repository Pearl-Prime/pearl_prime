SDD (Software Design Document) – Full Workflow
1. Problem & Goal
What are we building?
Who is it for?
What success looks like (clear outcomes).
2. Scope & Constraints
In-scope features.
Out-of-scope (explicitly).
Constraints: time, budget, platforms, policies.
3. Requirements
Functional: what the system must do.
Non-functional: performance, scale, security, reliability.
Acceptance criteria (how we know it works).
4. System Architecture
High-level diagram (inputs → processing → outputs).
Key components/services.
Data flow between components.
5. Data Design
Core data models / schemas.
Storage choices (DB, files, cache).
Validation and lifecycle rules.
6. API & Interfaces
Public/internal APIs.
Inputs, outputs, error cases.
Auth rules.
7. Core Logic & Algorithms
Business rules.
Edge cases.
Deterministic vs probabilistic behavior.
8. Dependencies & Integrations
External services/APIs.
Versioning and failure handling.
Mocking strategy.
9. Security & Compliance
Auth, permissions.
Secrets handling.
Logging & audit rules.
10. Testing Strategy
Unit tests.
Integration tests.
CI gates / fail conditions.
11. Deployment & Operations
Environments (dev/staging/prod).
Config & secrets.
Rollback strategy.
12. Risks & Open Questions
Known risks.
Unknowns to validate.
Future extensions (non-binding).

Golden Rule
If a dev can build it without asking questions, the SDD is done.
