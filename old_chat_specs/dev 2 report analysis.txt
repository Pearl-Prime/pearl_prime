# Writer Spec for 100% Parity (Omega V4)

**Purpose:** If writers follow this spec exactly, make validate-registry-parity passes and zh_tw builds stay in sync with EN.

**Audience:** Content team, translators, anyone editing section content.

---

## 1. Where to Edit

| Locale | Edit this | Do not hand-edit |
|--------|-----------|------------------|
| **EN** | phoenix_v4/libraries/sections/registry.yaml | Per-variant YAMLs (they are generated) |
| **zh_tw** | SOURCE_OF_TRUTH/data/locale/zh_tw/phoenix_v4/libraries/sections/registry.yaml | Per-variant YAMLs (they are generated) |

**Rule:** Registry is canonical. After any change to a registry, run the export script so per-variant files and fingerprints stay in sync:

bash
# After editing EN registry
python3 tools/export_locale_registry.py --locale en --update-registry

# After editing zh_tw registry
python3 tools/export_locale_registry.py --locale zh_tw --update-registry

If you must edit a per-variant YAML by hand (e.g. one-off fix), you must still follow §2–§5 below and then run export with --update-registry so the registry is updated to match.

---

## 2. Variant Block Schema (Exact)

Every variant must have exactly these keys. No extra keys. No renamed keys.

| Key | Type | Example | Rule |
|-----|------|---------|------|
| variant_id | string | ch01_sec01_hook_f1 | Format: ch + 2-digit chapter + _sec + 2-digit section + _ + type + _ + family. **No localized IDs** (e.g. no 第一章第一节-hook-f1). |
| chapter | integer | 1 | Same as in registry section. |
| section | integer | 1 | Same as in registry section. |
| section_type | string | HOOK, SCENE, REFLECTION, TEACHER_DOCTRINE, INTEGRATION, EXERCISE | Same as registry. |
| purpose | string | Recognition — the sentence reveal | Same as EN (zh_tw may translate). |
| variant_family | string | F1, F2, D01 | Same as registry. |
| content | string | See §3 | Prose only. No placeholders like [INTAKE INCOMPLETE]. |
| fingerprint | object | See §4 | Must match computed values from content. |
| metadata | object | persona, topic, scene_type, location_tokens | Same shape as registry. |

---

## 3. Content Rules

### 3.1 Topic placeholder (EN)

- Use **"the feeling"** as the topic-agnostic placeholder.
- Do **not** replace it with a specific topic (e.g. "anxiety", "stress") in the canonical EN source.

### 3.2 Topic placeholder (zh_tw)

- Use **one consistent placeholder** for "the feeling" across all zh_tw content (e.g. **那種感受** or **這種感受**). Document the choice in the project glossary.
- **Never** use 焦虑, 緊張, or any specific diagnosis/topic as the default translation of "the feeling."
- Topic-specific wording is allowed only where the EN source explicitly names a topic.

### 3.3 Forbidden in content

- [INTAKE INCOMPLETE]
- TODO
- TRANSLATE LATER
- Any other stub or placeholder that is not final reader-facing copy.

### 3.4 YAML safety (if editing per-variant YAML by hand)

- Content often contains colons (:) and newlines. To avoid parse errors:
  - **Preferred:** Store content as a **single double-quoted string** with \n for newlines (e.g. content: "First line.\n\nSecond para.").
  - **Alternative:** Use a YAML literal block (|) with consistent indentation; ensure no line starts with a word followed by :  or the parser will treat it as a key.
- Running the export script from the registry produces valid YAML automatically; hand-editing is error-prone.

---

## 4. Fingerprint (Exact Definition)

The validator recomputes fingerprint from content and compares to the stored value. They must match for 100% pass.

| Key | Definition |
|-----|------------|
| char_count | len(content) (number of characters). |
| paragraph_count | Number of non-empty blocks when splitting content by \n\n (blank line). Trim each block; count only non-empty. |
| sentence_count | Number of occurrences of . or ? or ! in content. If zero, use 1 if content is non-empty after strip, else 0. |

**Example (EN):**  
Content with 864 characters, 13 paragraph blocks, 13 sentence-ending punctuation marks →  
fingerprint: { char_count: 864, paragraph_count: 13, sentence_count: 13 }.

**Writers:** Prefer running export_locale_registry.py --update-registry so fingerprints are recomputed for you. If you edit content by hand, either run the export after editing or recompute using the rules above.

---

## 5. File Location (Per-Variant YAML)

If you add or move a variant, the file must live at:

{sections_root}/chapter_{NN}/section_{NN}_{type}/{family}.yaml

- NN = zero-padded 2-digit chapter or section number (e.g. 01, 06).
- type = section_type lowercased, with spaces replaced by _ (e.g. hook, teacherdoctrine).
- family = variant_family lowercased (e.g. f1, d01).

**Examples:**

| variant_id | Path |
|------------|------|
| ch01_sec01_hook_f1 | chapter_01/section_01_hook/f1.yaml |
| ch01_sec06_teacherdoctrine_d01 | chapter_01/section_06_teacherdoctrine/d01.yaml |
| ch12_sec07_integration_f1 | chapter_12/section_07_integration/f1.yaml |

**Writers:** You do not need to create these files by hand. Edit the registry and run the export script; it creates/overwrites them.

A V4 registry.yaml
V4 per-variant section YAMLs (chapter_XX/section_YY_type/*.yaml)
Only EN and zh_tw currently have that.
This means:
The main book body (V4 sections) is not translated for EU yet
The system falls back to EN for that layer
That fallback is expected and correct.

How V4 behaves for EU today
When you generate a V4 book with:


locale: de_de

The system does this:
Looks for SOURCE_OF_TRUTH/data/locale/de_de/phoenix_v4/libraries/sections/
Doesn’t find it → falls back to EN
Uses:
EN V4 section content (12-chapter body)
EU-translated legacy content where applicable (exercises, wrappers, A0/A1, etc.)
So EU output today is hybrid by design:
EN for V4 sections
EU for legacy layers

Why this clarification exists
This write-up exists to prevent a false assumption:
“EU languages must be missing work or broken.”
They’re not.
Instead, the system is saying:
EU was translated before V4 existed
V4 is newer and stricter
When EU gets V4 content, it must follow the new contract from day one
That’s why:
V4_LOCALE_INTAKE_CONTRACT.md exists
WRITER_SPEC_100_PERCENT.md exists
Validator/export now accept any locale code
This is future safety, not present work.

Does this mean more writing is needed now?
❌ No
You are not expected to translate V4 into EU now
There is no backlog implied
Nothing is blocking current output
✅ It does mean
When (and only when) you decide to localize V4 for EU:
You must add SOURCE_OF_TRUTH/data/locale/{locale}/phoenix_v4/libraries/sections/
You must include a translated V4 registry.yaml
You must run export and validation
You must follow the writer spec and intake contract
Until then, EN fallback is the official behavior.
