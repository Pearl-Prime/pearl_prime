## 1. Core principle

**Prefer real pipeline outputs.** Every onboarding example should be either:

1. **Production-equivalent output** — generated by the same scripts/pipelines that ship product, or  
2. **Pipeline demo run** — explicitly labeled in registry as `source: pipeline_demo` (same toolchain, fixed seed/config), never arbitrary stock art pretending to be product.

The **registry** is the contract surface; this spec defines **what rows must exist** and **how generations are batched and approved**.

---

## 2. Three visual categories (do not mix without labeling)

| Category | Role in onboarding | Registry `format` examples | Source |
|----------|----------------------|----------------------------|--------|
| **A — Real pipeline outputs** | Prove “what the system ships” | `cover`, `interior_page`, `article_hero`, `manga_panel`, `landing_hero`, `audiobook_package` | Pipeline / export jobs |
| **B — Audience / persona visuals** | Teach “who we write for” | `persona_visual`, `audience_card` | Curated photo, illustration, or **image pipeline** with fixed brief; label `source` honestly |
| **C — Topic / emotion visuals** | Teach thematic worlds (burnout, hope, grief) | `topic_visual`, `emotion_scene` | Same as B — must not masquerade as book cover output unless it is one |

**Rule:** Categories B and C are allowed for empathy and clarity, but registry entries **must** use formats and captions that state they are **supporting visuals**, not fake book/manga deliverables. Category A must be **indistinguishable from production path** (or marked `pipeline_demo`).

**Normalize in registry:** Every row **must** set `production_fidelity` (see §8) so filters never infer fidelity from `format` alone. Every row **must** set `proof_intent` so page logic knows whether the tile is product proof, empathy teaching, or a comparison-board member.

## 3. Minimum example matrix (smallest set that teaches differentiation)

Goal: **not** infinite grids — the **minimum** set that lets a brand admin see *how choices change output*.

### 3.1 By lane (minimum slots)

| Lane | Required proof types (Category A) | Teaching intent |
|------|-------------------------------------|-----------------|
| Self-help | Cover + 1 interior spread + 1 landing hero | Packaging + reading experience |
| Audiobook | Cover + chapter/preview packaging + 1 “listening” promo card | Audio-adjacent merchandising |
| Manga / visual story | Cover + 3–6 panel page + 1 promo thumbnail | Visual narrative + serialization feel |
| Pearl News / civic editorial | Article hero + story card + short layout snippet | Editorial trust + scanability |
| Teacher / educational mode | Lesson-style page or card (per product definition) | Clarity + structure |
| Breathwork / DELF / tools | Tool UI screenshot + 1 landing hero | Utility + funnel entry |
| Youth civic explainer | One explainer card or short article visual | Tone + responsibility |

Each slot = one or more registry rows (e.g. locale variants = separate rows).

### 3.2 By variation type (comparison sets)

Onboarding must include **controlled comparisons** — same underlying **persona + topic**, varied along one axis:

| Variation axis | Example | Registry grouping (align with §3.3) |
|----------------|---------|-------------------------------------|
| Persona | Burned-out founder vs anxious student | `cmp_burnout_cross_persona_v1` |
| Market / locale | US EN vs JP vs TW (same topic/persona) | `cmp_anxiety_locale_v1` |
| Format | Self-help cover vs manga page vs article hero | `cmp_burnout_format_v1` |
| Tone / brand posture | Premium spiritual vs practical mainstream | `cmp_burnout_posture_v1` |
| Manga sub-mode | Cinematic vs civic vs healing | `cmp_manga_style_mode_v1` |

Additional axes (e.g. **topic** variation at fixed persona) may use new `cmp_*` ids in a later v2; v1 is fully defined by §3.3.

### 3.3 Locked first comparison sets (v1 contract)

These **`comparison_set_id` values are required in the registry plan** (rows may start as `status: "planned"`). They are the **authoritative v1 set** for teaching “one variable at a time.” UI and gallery modules should key off these IDs.

| Priority | `comparison_set_id` | Varying axis | Fixed context (illustrative) | Member rows must share |
|----------|---------------------|--------------|------------------------------|-------------------------|
| 1 | `cmp_burnout_cross_persona_v1` | Persona | Topic: `burnout` | Same `topic`, `market`, `format` family; different `persona` (e.g. founder vs student vs caregiver) |
| 2 | `cmp_anxiety_locale_v1` | Market / locale | Topic: `anxiety`, one persona | Same `persona`, `topic`; different `locale` / `market` (e.g. en-US, ja-JP, zh-TW) |
| 3 | `cmp_burnout_format_v1` | Format / lane output type | Persona + topic fixed | Same `persona`, `topic`; different `format` / `lane` (e.g. self-help cover vs manga panel vs article hero) |
| 4 | `cmp_burnout_posture_v1` | Brand posture | Persona + topic fixed | Same `persona`, `topic`, `format`; different `style_variant` or posture knob (e.g. premium_spiritual vs practical_mainstream) |
| 5 | `cmp_manga_style_mode_v1` | Manga sub-mode | One topic + persona | Same `lane: manga`; different `style_variant` (see §4: cinematic, youth_inspire, spiritual, healing, civic) |

**Rollout:** Tier implementation as **1 → 2 → 3** first if bandwidth is limited; still **register** all five sets with planned rows so the schema and UI do not churn. **Marketing “example-complete”** (internal bar): at least **three** of the five sets have **all** member tiles `status: "ready"` **or** an explicit written exception in QC docs.

Until rows are ready, `status: "planned"` / `"missing"` and proof-pending UI apply per scaffold plan.

## 4. Manga differentiation spec (not one bucket)

Manga onboarding must expose **sub-modes** with distinct registry slots (examples — adjust names to match pipeline taxonomies):

| Manga style variant | `style_variant` example | What the example must show |
|---------------------|-------------------------|-----------------------------|
| Cinematic / high dynamism | `manga_cinematic` | Action-forward panels, motion |
| Youth inspiration | `manga_youth_inspire` | Bright, social-native energy |
| Spiritual / philosophical | `manga_spiritual` | Symbolic restraint, contemplative |
| Emotional healing | `manga_healing` | Soft pacing, body/emotion cues |
| Issue-driven / civic | `manga_civic` | Clarity, dignity, non-exploitative |

Each variant: **at least** cover + panel strip + promo thumbnail (Category A), or `status: "missing"` until generated.

For **global** onboarding, prioritize rows for: `us`, `japan`, `taiwan`, plus **one** “global English” lane — expand per product roadmap.

Registry should plan rows for (IDs illustrative): `anxious_student`, `burned_out_founder`, `overwhelmed_caregiver`, `spiritual_seeker`, `youth_civic` — not all need `status: "ready"` at once.

## 6. Generation contract (pipeline ↔ registry)

### 6.1 Naming and versioning

- **Stable IDs:** `id` in registry is immutable once published; new visuals get new IDs or a bumped `version` field.  
- **Paths:** `asset_path` relative to repo or CDN base documented in deployment spec (e.g. under `artifacts/onboarding_examples/` or Cloudflare Pages static).  
- **Provenance:** Every row includes `source`, `created_at`, `version`; pipeline jobs should append a run id or commit SHA in a `generation_run` field (optional) for audit.

### 6.2 Batch workflow (recommended)

1. **Plan** — Add/update registry rows with `status: "planned"` and `comparison_set_id` where applicable.  
2. **Generate** — Run documented pipeline commands (Pearl_Dev / visual pipelines — exact commands TBD per subsystem).  
3. **QC** — Human approval: on-brand, no misleading claims, correct locale, accessible contrast where applicable.  
4. **Publish** — Set `status: "ready"`, upload assets, verify URL resolves in staging Pages.  
5. **Stale detection** — If asset 404s, UI falls back to proof-pending (see scaffold plan); ops should set `status` back to `"planned"` or fix asset.

### 6.3 When a batch is incomplete

- Do **not** block onboarding releases: leave `status: "missing"` or `"planned"`.  
- Optional: `asset_exists: false` for ops tracking; **UX always follows `status`**, per scaffold plan.

---

## 7. QC and approval rules (minimum)

- **Accuracy:** Caption must match what the image is (no “cover” label on a persona stock shot).  
- **Consent / likeness:** Persona/topic imagery: no deceptive “real customer” implications.  
- **Locale:** Text in frame matches `locale` and market expectations.  
- **Brand safety:** No slurs, exploitative civic imagery, or medical claims beyond product’s therapeutic positioning.  
- **Manga:** Civic and youth-facing variants get extra pass for tone and dignity.

## 8. Registry schema (normalized fields)

Each requirement in this spec maps to one or more **registry rows**. Below is the **contract** for onboarding proof rows in `config/onboarding/example_registry.json`.

### 8.1 Required fields (every row)

| Field | Type / values | Notes |
|-------|-----------------|--------|
| `id` | string | Stable id; immutable once published to prod UI |
| `lane` | string | e.g. `self_help`, `manga`, `pearl_news`, … |
| `market` | string | e.g. `us`, `japan`, `taiwan` |
| `locale` | string | e.g. `en-US`, `ja-JP`, `zh-TW` |
| `persona` | string | Align with catalog / `unified_personas.md` where possible |
| `topic` | string | Align with catalog / `canonical_topics` where possible |
| `format` | string | Visual shape: `cover`, `manga_panel`, `persona_visual`, … |
| `style_variant` | string | Posture, manga mode, packaging lane, etc. |
| `status` | `"ready"` \| `"planned"` \| `"missing"` | **Primary UX driver** for scaffold UI (see onboarding stack plan) |
| `proof_intent` | enum (required) | See §8.2 |
| `production_fidelity` | enum (required) | See §8.3 |
| `product_family` | enum (required) | See §8.4 |
| `source` | string | e.g. `pipeline`, `pipeline_demo`, `curated` |
| `asset_path` | string | Relative path or CDN key; may be empty when not `ready` |
| `caption` | string | Human-readable; must match actual asset type |
| `version` | string | e.g. `v1` |
| `created_at` | string | ISO date |

### 8.2 `proof_intent` (required)

Controls **why** the tile exists for onboarding logic (filters, layout, copy). Two rows can share `format: persona_visual` but differ here.

```text
proof_intent: "ships_product" | "teaches_persona" | "teaches_topic" | "teaches_comparison"
```

| Value | Use when |
|-------|----------|
| `ships_product` | Showing real deliverable output (Category A; cover, article hero, manga page, tool UI, …) |
| `teaches_persona` | Audience empathy card; not claiming to be a shipped book cover |
| `teaches_topic` | Topic / emotion world; not a product mock unless it is pipeline output |
| `teaches_comparison` | Member of a comparison board; same `comparison_set_id` as siblings |

Comparison-set members should use `teaches_comparison` (and usually `production_fidelity: production` or `pipeline_demo` for pipeline tiles, or `supporting_visual` only if the comparison is explicitly about non-product visuals).

### 8.3 `production_fidelity` (required)

Normalized truth for **how close to shipped product** the asset is. Do not infer this only from §2 categories.

```text
production_fidelity: "production" | "pipeline_demo" | "supporting_visual"
```

| Value | Meaning |
|-------|---------|
| `production` | Same path and fidelity as customer-facing output |
| `pipeline_demo` | Real toolchain, fixed seed/config; labeled for onboarding |
| `supporting_visual` | Persona/topic/emotion support; not a product deliverable |

### 8.4 `product_family` (required)

Links the row to the **generating product system** for handoffs and filters.

```text
product_family: "pearl_news" | "book_engine" | "manga" | "audiobook" | "tool_ui" | "funnel_landing" | "teacher_mode" | "other"
```

Extend `other` only with a spec note; prefer adding a new enum value if a family becomes recurring.

### 8.5 Optional fields

| Field | Use |
|-------|-----|
| `comparison_set_id` | Required for any row in §3.3 sets; groups board tiles |
| `visual_category` | `pipeline_output` \| `persona` \| `topic` — mirrors §2 for reporting |
| `asset_exists` | Boolean; ops only; **never** override `status` for UX |
| `generation_run` | Pipeline run id or commit SHA |
| `qc_approved_by` | Optional human gate |

## 10. Summary

This spec turns the **strong registry model** into a **product contract**: which real (or honestly labeled) examples must exist, how they differ by lane and manga mode, how comparison sets teach decisions, and how partial delivery still works with `status` + proof-pending UI. Implementing it is a **Pearl_Dev / pipeline + Pearl_Editor QC** program, not a one-off design task.

---

## 11. Handoffs (who does what)

| Team | Responsibility |
|------|----------------|
| **Product** | Owns comparison-set priorities, “example-complete” bar, and sign-off that captions match positioning (no misleading product claims). |
| **Pearl_Dev / pipeline** | Implements generation jobs per `product_family`; writes `generation_run`; keeps `production_fidelity` accurate. |
| **Visual / manga pipeline** | Owns Category A assets for manga and cover-adjacent outputs; maps `style_variant` to pipeline configs. |
| **Pearl_Editor / QC** | Approves rows for `status: "ready"`; flags stale assets; enforces §7. |
| **Pearl_GitHub / integration** | Deploy-bound JSON and static assets per onboarding Cloudflare plan; secrets and non-blocking vs required checks. |
