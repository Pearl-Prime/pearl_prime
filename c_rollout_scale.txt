Assessed technical specifications for multiple web components

The architecture is correct. One template per brand, parameterized per book, static hosting, single Apps Script backend — that's the right design. A few things to flag:

The Apps Script has a real bug. The scheduleFollowup function creates a new time-based trigger for every single email capture. Google Apps Script has a limit of 20 triggers per script. After 20 freebie downloads, your follow-up system silently breaks. The fix is straightforward: instead of per-capture triggers, run a single daily cron trigger that scans the Google Sheet for rows where followup_sent == false and created_at is older than 7 days, then sends and marks them. One trigger, unlimited scale.

The PropertiesService approach won't scale either. Script properties have a total storage limit of 500KB. Storing full JSON blobs per capture will hit that ceiling fast. Since you're already writing to a Sheet, just read from the Sheet in your daily cron. The Sheet is your database — don't duplicate state in properties.

The landing page HTML is too bare. It works mechanically, but a system-ui font on a white page with no brand identity will feel like a phishing page to someone who just finished a beautifully voiced somatic healing session. Each brand template needs at minimum the brand's color palette from your cover art identity spec, the teacher name displayed (not just in a hidden field), and a one-line description pulled from the book's emotional vocabulary. Your generator already has all this data — it just needs to inject it into the template. This doesn't mean 24 bespoke designs. It means one HTML template with CSS variables that get populated per brand at generation time.

The URL scheme choice matters more than the doc suggests. Go with the pretty version (/free/{brand_id}/companion/{book_id}/) and generate actual index.html files in those directories. The query-string version (?book_id=...) means your hidden form fields depend on JavaScript parsing URL params. If JS fails to load or a privacy browser strips query strings, your form submits with empty metadata and you get useless captures. Static files with baked-in values are more reliable and cost nothing extra since your generator produces them anyway.

Missing: unsubscribe handling. Any automated email system — even a single follow-up — needs a way to opt out. CAN-SPAM requires it, and Google will eventually flag your sending domain if people mark follow-ups as spam because there's no unsubscribe link. Add a simple "Don't want more emails? Reply STOP" line to the follow-up template, and have your daily cron check a "stopped" column before sending.

The generator extension output is clean. Landing URL, download URL, and related titles — that's the right data contract. The assembler only sees landing_url, everything else stays downstream. Good separation.

So: fix the trigger architecture (daily cron, not per-capture), drop the PropertiesService hack, brand the templates with CSS variables, use static file paths instead of query strings, add unsubscribe handling, and it's ready to build.
